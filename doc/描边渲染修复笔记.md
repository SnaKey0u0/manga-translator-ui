# 描边渲染修复笔记

> 2026-05-13 · 记录一次描边脏边问题的根因分析、最终修复、以及途中踩到的坑。

## 问题现象

后端渲染出来的文字描边很脏 —— 在抗锯齿边缘出现灰色过渡像素、warp 变换后出现黑色晕边。编辑器预览却没这个问题。

## 根因分析

共有 4 个独立问题叠加：

### 1. 灰边（混合模型错误）

`add_color` 里把文字和描边视作两个独立的半透明图层做 Porter-Duff over 合成：

```
out_rgb = (fg_rgb * fg_a + bg_rgb * bg_a * (1 - fg_a)) / out_a
```

在抗锯齿过渡像素上（比如文字 α=130、描边 α=100），上式会算出**中间灰** —— 这是数学必然。

### 2. 描边对不齐文字（光栅化偏差）

文字路径和描边路径分别交给 Qt 光栅化成两张 alpha 图。抗锯齿算法的微小差异会让描边 alpha 在某些边缘像素 **小于** 文字 alpha，出现"文字有值但描边没盖住"的情况，配合问题 1 就会露出深灰脏点。

### 3. warp 后出现黑/灰晕边（插值光晕）

`cv2.warpPerspective` / `cv2.resize` 是**非预乘** RGBA 插值。边缘像素会和"透明黑" `(0,0,0,0)` 混合，结果把 RGB 向黑色拉，产生黑色晕边。

### 4. uint8 溢出隐患

在描边 alpha 和文字 alpha 直接 `a + b` 或 `np.add` 的场景下，两个 130 相加会变成 4（wrap），呈现为亮区中突然的黑斑。

## 最终修复

### `manga_translator/rendering/text_render.py::add_color`

改变合成模型：把描边作为**文字的底色**，让文字抗锯齿过渡像素直接在描边纯色上混合，而不是两个半透明层叠加。

关键改动：

1. **强制 `stroke_alpha = max(stroke_alpha, text_alpha)`**，保证描边在空间上完全覆盖文字的所有抗锯齿像素，消除对齐偏差。
2. 合成公式从 `fg*a + bg*b*(1-a)` 改成 `fg*a + bg*(1-a)`，bg 作为不透明底色。
3. 全程 float32 + np.clip，彻底避免 uint8 溢出。

### `manga_translator/rendering/__init__.py` 合成链路

在把 RGBA 文本位图做 `warpPerspective` 之前，先做 **Alpha 预乘**：

```python
box_pm[:, :, c] = box[:, :, c] * (box[:, :, 3] / 255.0)
```

warp 之后做**预乘 src-over** 合成：

```python
out = target * (1 - src_alpha) + src_premul
```

预乘语义下插值不会和透明黑混合，黑/灰晕边彻底消失。

### `manga_translator/rendering/text_render_hq.py`

HQ 模式里 `cv2.resize` 也是插值，同样做"预乘 → resize → 反预乘"三步，保证输出给下游的还是非预乘 RGBA（兼容旧接口）。

## 踩的坑（重构方案的回退）

最初我考虑过一个更激进的重构：把"两次独立光栅化 alpha + numpy 手工合成"改成"Qt 一次 QPainter pass 同时 stroke + fill，原生输出预乘 RGBA"。理论上这条路最干净，最终还是**全部回退**了。记录下踩到的三个坑，避免以后有人再走一遍：

### 坑 1 — Qt drawPath 的 stroke/fill 顺序

```python
painter.setPen(pen)     # stroke 颜色 = bg
painter.setBrush(brush) # fill   颜色 = fg
painter.drawPath(path)  # 一次搞定？
```

**错了。** `drawPath` 内部是**先 fill 再 stroke**，线宽 `stroke_px * 2` 完全覆盖了填充 —— 整张图只有白色描边，没有黑色文字。

正确做法是拆成两次 drawPath：先只 stroke（NoBrush），再只 fill（NoPen），让填充盖在描边内半。

### 坑 2 — 验证脚本的不变式选错

我的验证脚本检查"最不透明像素离中灰 (128,128,128) 的 L∞ 距离"，以为只要**有**远离中灰的像素就说明渲染对了。结果纯白也距中灰 127 —— 验证通过，实际只画出了白色描边。**不变式要基于行为差异对比（改动前后像素）**，不要选自己想得到的属性。

### 坑 3 — 双模函数的传参链路

重构时让 `_line_surface` / `_block_surface` 兼任两个职责：`fg=None` 时纯测量，提供 `fg` 时彩色光栅化。表面上省代码，但：

- `_build_vertical_layout` 调 `_block_surface` 时忘记透传 fg/bg → 渲染时走了测量分支 → `rgba=None` → 下游 `continue` → **竖排里的 `<H>` 横排块整个不渲染**。
- `_block_surface` 没处理测量分支返回的 `rgba=None` → `rgba.shape` 抛 NoneType。

双模函数是脆弱设计。如果将来真要走这个方向，应当拆成两个独立函数（`_line_measure` / `_line_rgba`），用签名强制调用方正确使用。

## 后续可考虑的方向

当前修复属于"**最小侵入，合成语义正确**"。如果未来要进一步简化，可以考虑：

1. 把 `_paste_glyph_pair` 的竖排 `add` 模式改 `max`，和横排 `_paste_surface` 统一规则（轻微改变竖排字间的描边接合视觉，建议单独分支验证）。
2. 把 `_rasterize_path` + `_stroke_path` 合并为"一次 QPainter 先 stroke 后 fill"输出预乘 RGBA 的函数。下游链路全部改走预乘（渲染、warp、resize、合成）。技术上可行，但涉及面广，需要完整的回归测试覆盖 —— 竖排 `<H>` 块、竖排单字符、HQ 放大、编辑器预览、warp 变换。

以上两条都不是本次修复的范围。本次只做正确性修复。

## 2026-05-14 续：描边外缘 #f9f9f9 残留灰

之前的修复让 `add_color` 输出的描边像素是纯白 `(255,255,255)`，预乘前也是纯白。但渲染到原图后，描边外缘仍然有约 5% 的灰（≈#f9f9f9），系统性地出现在所有文本块上。

### 根因 —— Lanczos4 在预乘空间的振铃

`rendering/__init__.py` 里 warpPerspective 用的是 `cv2.INTER_LANCZOS4`。Lanczos4 是 sinc 加窗的插值核，**有负 lobe**。在预乘 RGBA 上做透视变换时：

- 描边外缘是 `(255,255,255,180)` → `(0,0,0,0)` 的陡峭过渡
- 4×4 邻域加权时负 lobe 把"远处的透明黑"按负权重减出来 → 过渡区预乘 RGB 被压低
- α 通道也有振铃，但因为 dilate 后的 α 边界比 RGB 边界更"硬"，**两者振铃量不成比例**
- 反预乘 `rgb / alpha × 255` 把这个不成比例放大到肉眼可见的 ~5% 灰

诊断日志关键证据（A=[50,199] 桶）：

| 阶段 | RGB mean |
|------|----------|
| `add_color/out` | (255, 255, 255) |
| `pre-warp/box` (非预乘) | (255, 255, 255) |
| `pre-warp/box_pm` (预乘后) | RGB ≈ alpha 值，符合预期 |
| `post-warp/rgba_region` 反预乘 | **(241.6, 241.7, 241.5)** ← 灰来自这一步 |

### 修复

`rendering/__init__.py::dispatch` 把 `cv2.INTER_LANCZOS4` 换成 `cv2.INTER_LINEAR`。双线性无负 lobe，预乘空间下不会振铃。漫画文字大字号缩放下，肉眼看不出和 Lanczos4 的锐度差异。

### 备注

- HQ 模式 `text_render_hq.py` 用的是 `cv2.INTER_AREA`（缩小专用），同样无负 lobe，不需要改。
- 如果将来发现某些场景对锐度敏感（极小字号放大），可以考虑 `INTER_CUBIC` —— 负 lobe 比 Lanczos4 小很多，但仍存在，需评估灰边量级。
